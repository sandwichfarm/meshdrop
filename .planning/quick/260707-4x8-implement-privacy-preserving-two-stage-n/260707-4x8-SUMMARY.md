# Summary: Privacy-Preserving Two-Stage Nostr Route Discovery

## Changed

- Public Nostr Mesh presence now publishes minimal session presence: MeshDrop/WebRTC capability, random session peer id, expiration, and FIPS/Pollen route capability flags only.
- Default WOT FIPS/Pollen enable no longer pre-joins every pairwise npub-network room. Explicit public/debug rooms can still join immediately; private pairwise rooms now come from route descriptors.
- Direct Nostr WebRTC remains the first route. When it fails and no fallback candidate is already present, the peer manager requests an encrypted route descriptor over Nostr.
- Nostr route descriptor exchange now uses NIP-44 `route-request` and `route-response` bodies with nonce, session id, recipient, route type, and expiry validation before local route rooms are joined.
- FIPS and Pollen controllers expose one-peer descriptor generation and descriptor join helpers without publishing HTTP bases, Pollen services, device names, or platform fields in WOT presence.

## Risks

- Route descriptors currently carry npub-network room ids only. There is no TURN/relay-only ICE transport descriptor yet, so WebRTC media/data path can still use clearnet ICE unless runtime ICE policy changes in a later transport-relay slice.
- Pending private route requests do not currently have a UI-visible timeout beyond route status `requested`; lack of response leaves the peer waiting instead of immediately disconnecting.

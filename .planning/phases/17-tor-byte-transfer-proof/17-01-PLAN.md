# Phase 17 Plan: Tor Byte Transfer Proof

## Goal

Prove a Tor overlay stream route transfers real payload bytes through a Dockerized `.onion` endpoint, with hash-matched route proof and no Clearnet fallback claim.

## Scope

- Add generic overlay stream transfer storage/descriptor code for backend overlay adapters.
- Wire server endpoints for uploading/downloading configured overlay stream payloads.
- Add tests for descriptor validation, token/expiry/size behavior, and runtime capability boundaries.
- Add `npm run test:tor-stream` that starts a reproducible Tor hidden-service path in Docker and fetches the payload through the onion route.
- Update ADR/docs and issue #151 with exact Tor proof while keeping I2P/Loki blocked.

## Result

Complete on 2026-07-07.

- Added generic overlay stream transfer storage and HTTP endpoints under `/overlay/:route/*`.
- Added `npm run test:tor-stream`, which builds a Tor test image, starts a Tor hidden service, configures MeshDrop with the generated `.onion` endpoint, uploads a payload, fetches it through SOCKS with remote hostname resolution, and validates route proof.
- Added ADR 0009 for the non-WebRTC `tor-http-stream` proof boundary.
- Kept I2P/Loki blocked under issue #151 pending equivalent daemon/proxy proof.

## Risk

- Tor Docker image behavior and bootstrap timing can be slow or flaky; the smoke must wait deterministically and fail with logs.
- Browser WebRTC over Tor is not in scope; this phase proves a non-WebRTC `tor-http-stream` route.
- I2P/Loki remain blocked until equivalent route-specific harnesses exist.

# Phase 14 Plan: Overlay Network Adapters

## Goal

Add Tor, I2P, and Loki as first-class backend route adapter types that use the shared runtime capability and route descriptor vocabulary, while failing closed until a local instance/native runtime can prove it can dial those networks.

## Requirements

- ONA-01
- ONA-02
- ONA-03
- ONA-04
- ONA-05

## Plan

1. Add failing focused tests for Tor/I2P/Loki runtime capability defaults, configured server adapters, static SPA fail-closed behavior, and generic route descriptor/scoring acceptance.
2. Add a server-side overlay adapter catalog that normalizes Tor, I2P, and Loki config through one path and emits stream metadata only when explicitly configured.
3. Wire overlay adapter capabilities into server `/config` and browser static runtime config without changing existing FIPS/Pollen capabilities.
4. Keep Tor/I2P/Loki out of live Nostr route capability publication until a descriptor response and byte-transfer primitive exist.
5. Verify with focused tests, broad tests, browser/docker smokes, diff check, and changed-code slop scan.

## Acceptance

- Default `/config` capability entries for `tor`, `i2p`, and `loki` are present, unsupported, backend-required, and explicit about why they are unavailable.
- Configured server capability entries use `transportShape="stream"` and route-specific primitives: `tor-http-stream`, `i2p-http-stream`, and `loki-http-stream`.
- Backend-free SPA config refuses Tor/I2P/Loki support even when static target metadata claims those transports.
- Generic route contract tests validate and score Tor/I2P/Loki descriptors as ordinary private stream candidates.
- No route proof claims Tor/I2P/Loki byte-transfer completion in this phase.
